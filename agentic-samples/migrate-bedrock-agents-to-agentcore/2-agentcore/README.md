# Phase 2 — Amazon Bedrock AgentCore Runtime

This directory contains the AgentCore Runtime version of the private aviation agent, generated by importing the Phase 1 Bedrock Agent using the AgentCore CLI.

## Architecture

![Phase 2 architecture: caller invokes AgentCore Runtime via the HTTP contract; the container runs a Strands agent with four in-process tools (current_time, get_owner_info, get_passengers, book_reservation) that call Amazon Bedrock directly; observability is auto-instrumented](docs/architecture.png)

[Amazon Bedrock AgentCore](https://aws.amazon.com/bedrock/agentcore/) Runtime inverts the contract from Phase 1. Instead of "configure tools and AWS builds the agent," the rule is: build the agent with any framework, AgentCore runs it. The agent is a standard HTTP server that implements `POST /invocations` (prompts) and `GET /ping` (health). Runtime handles scaling, networking, IAM, and observability around that contract. The sample uses [Strands Agents](https://github.com/strands-agents/sdk-python), but the same HTTP contract accepts LangGraph, PydanticAI, Google ADK, and others.

Two deployable variants ship in this phase, covered in [Two entrypoint variants](#two-entrypoint-variants) below. The `main.py` default inlines the three tools as `@tool` functions inside the Strands container. The `main_gateway.py` variant preserves the Phase 1 [AWS Lambda](https://aws.amazon.com/lambda/) functions by placing them behind an AgentCore Gateway and having the agent call the Gateway as an MCP client.

## How this was generated

After deploying the Bedrock Agent (see `../1-bedrock-agents/`), the import command reads the agent's configuration and scaffolds a Strands project:

```bash
agentcore create --type import \
  --name PrivateAviationAgent \
  --agent-id <AGENT_ID> \
  --agent-alias-id <ALIAS_ID> \
  --region us-east-1 \
  --framework Strands \
  --defaults
```

The CLI translates each `functionSchema` action group into a `@tool` Python function stub with the right type hints and docstring, sets up the Strands `Agent()`, and emits the `BedrockAgentCoreApp` entrypoint. Stub bodies are marked `# TODO: Implement the logic for this action group function.` — a developer fills in the business logic (in this repo, it uses the same in-memory fixtures as Phase 1 for a self-contained demo).

The `PrivateAviationAgent/` directory is a snapshot of the generated output with the three tool bodies filled in.

**Note on the model configuration:** the CLI generator sets both `temperature=1` and `top_p=1` by default. Claude Sonnet 4.6 rejects both at once (`temperature and top_p cannot both be specified for this model`), so the snapshot drops `top_p`. If you regenerate via `agentcore create --type import`, apply the same fix.

## Two entrypoint variants

This phase ships two working `main*.py` files. Pick whichever matches your migration goal:

| File | Tool dispatch | Use when |
|---|---|---|
| `app/PrivateAviationAgent/main.py` (default) | Tools run **in-process** as `@tool` functions inside the Strands container | Greenfield, or you intend to retire the Phase 1 Lambdas |
| `app/PrivateAviationAgent/main_gateway.py` | Tools dispatch **over MCP** to an AgentCore Gateway that wraps the unchanged Phase 1 Lambdas | You want to preserve existing Lambdas, IAM, CI/CD, on-call pages during the first migration step |

To activate `main_gateway.py`, point the entrypoint at it in `agentcore/agentcore.json` and set the Gateway environment variables (`GATEWAY_MCP_URL`, `GATEWAY_CLIENT_ID`, `GATEWAY_CLIENT_SECRET`, `GATEWAY_TOKEN_ENDPOINT`, `GATEWAY_SCOPE`). The Gateway and its three Lambda targets are created the same way `3-agentcore-harness/Makefile` creates them — use `agentcore add gateway` + `agentcore add gateway-target` from that Makefile as a non-interactive reference.

The MCP-via-Gateway variant costs a little latency (one Gateway hop plus the Lambda cold start the `@tool` variant eliminates) and a little money (see the [AgentCore Gateway pricing page](https://aws.amazon.com/bedrock/agentcore/pricing/)). In return, the three Phase 1 Lambda functions stay in place — only their handler code changes, because Gateway does not translate MCP into the Bedrock Agents action-group envelope. The handler reads a flat input map instead of `event["parameters"]`, and returns a plain JSON response that matches the tool's `outputSchema` instead of the `{"messageVersion": "1.0", ...}` wrapper. See [Configure AWS Lambda as a target for your gateway](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/gateway-add-target-lambda.html) for the exact event and context shape.

## Prerequisites

- `@aws/agentcore` CLI (latest or preview)
- Python 3.10 or later, `uv` for dependency management inside the project
- `aws-targets.json` populated with your own account ID (see below)

## Before you deploy

The committed `agentcore/aws-targets.json` carries the placeholder account ID `123456789012`. Replace it with your own before running `agentcore deploy`:

```bash
cd PrivateAviationAgent
sed -i '' "s/123456789012/$(aws sts get-caller-identity --query Account --output text)/" agentcore/aws-targets.json
```

On Linux, drop the empty `''` after `-i`.

## Deploy

```bash
cd PrivateAviationAgent

# Install the generated CDK project's npm deps (first time only)
( cd agentcore/cdk && npm install )

agentcore deploy --yes
```

## Regenerate local manifests

The committed `agentcore/agentcore.json` is a snapshot of the CLI-generated scaffold with your author's ARNs stripped out. `agentcore deploy` repopulates it with ARNs from your own account. To regenerate the scaffold from scratch against your own Phase 1 Bedrock Agent, re-run the import:

```bash
agentcore create --type import \
  --name PrivateAviationAgent \
  --agent-id <YOUR_AGENT_ID> \
  --agent-alias-id <YOUR_ALIAS_ID> \
  --region us-east-1 \
  --framework Strands \
  --defaults
```

Then fill in the `# TODO: Implement the logic for this action group function.` stubs and redeploy.


## Clean up

AgentCore CLI v1.0.0-preview does not ship an `agentcore destroy` subcommand yet. Tear down the stack via CDK directly:

```bash
cd PrivateAviationAgent/agentcore/cdk
npx cdk destroy
```

`cdk destroy` removes the AgentCore Runtime, IAM execution role, ECR repository, and container build pipeline. The CloudFormation stack is named `AgentCore-PrivateAviationAgent-default`.
