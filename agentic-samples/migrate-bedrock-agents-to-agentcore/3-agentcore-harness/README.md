# Phase 3 — Amazon Bedrock AgentCore Harness

The same private-aviation agent from `1-bedrock-agents/`, rebuilt on [Amazon Bedrock AgentCore](https://aws.amazon.com/bedrock/agentcore/) Harness. No `@tool` functions, no `BedrockAgentCoreApp`, no orchestration code. `harness.json` declares the agent and the AgentCore CLI deploys it.

## Architecture

![Phase 3 architecture: caller invokes AgentCore Harness with per-call overrides (model, tools, system-prompt); the managed loop runs in a Firecracker microVM with harness.json declaring the model, system prompt, and tool references; the agentcore_gateway tool resolves to three MCP targets that front the Lambda functions; AgentCore Memory, Identity, and Observability are wired in by config](docs/architecture.png)

AgentCore Harness is a managed agent runtime with a clear contract: **defaults at creation time, overrides at invocation time.** Declare the model, tools, instructions, memory, and environment once in `harness.json`; override any of them on a single call without redeploying. Every session runs in its own secure, isolated [Firecracker](https://firecracker-microvm.github.io/) microVM. Shell access to the runtime session container is available through `InvokeAgentRuntimeCommand` (see [Run commands on the environment](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/harness-environment.html)).

Components:

- **3 Lambda functions**, same business logic as `1-bedrock-agents/`, with the Gateway MCP event shape (flat arguments on `event`), not the Bedrock Agents action-group envelope.
- **AgentCore Gateway**, exposes the Lambda functions as MCP tool targets. Added via `agentcore add gateway` and `agentcore add gateway-target`.
- **AgentCore Harness**, declarative agent scaffolded by `agentcore create`. Tools and model bind at invoke time; defaults live in `harness.json`.

> This sample does not use AgentCore Memory. Add a `memories[]` entry to `agentcore.json` and a `memory` reference to `harness.json` when your agent has a reason to remember across turns.

## Tooling

The AgentCore CLI (`@aws/agentcore@preview`, v1.0.0-preview or later) is the canonical path for Harness. `agentcore create --model-provider bedrock` routes to the harness scaffold. `agentcore deploy` runs CDK for IAM and container builds, then makes the `CreateHarness` and `CreateGateway` API calls in a post-CDK phase. The Makefile here wraps the CLI non-interactively so the whole deploy chain runs from `make deploy`.

Lambda functions are not created by the CLI. They exist independently of the Harness as the tool backends. This repo's `scripts/` directory contains the boto3 code that packages and deploys the three private-aviation Lambda functions.

## Naming

The scaffold uses `PvtAviation` as the project name. The Harness resource
in AWS is named `PvtAviation_PvtAviation` (project prefix + harness name),
which must satisfy `[a-zA-Z][a-zA-Z0-9_]{0,39}` — hence the short project
name. Gateway target names use hyphens (`get-owner-info`, `get-passengers`,
`book-reservation`) because the CDK template requires target names to
match `^([0-9a-zA-Z][-]?){1,100}$`.

## Prerequisites

- AWS CLI credentials for an account with AgentCore Harness preview access
- Python 3.10 or later, `pip install boto3`
- Node.js 18 or later with `@aws/agentcore@preview`: `npm install -g @aws/agentcore@preview`
- AWS CDK bootstrapped in the target account and region
- Region: `us-east-1`, `us-west-2`, `ap-southeast-2`, or `eu-central-1` (default: `us-east-1`, override with `AWS_REGION=...`)

## Before you deploy

Three committed files ship with placeholder AWS account ID `123456789012` and placeholder ARNs:

- `project/PvtAviation/agentcore/aws-targets.json`
- `project/PvtAviation/agentcore/agentcore.json`
- `project/PvtAviation/app/PvtAviation/harness.json`
- `config/state.json`

Only `aws-targets.json` needs a manual edit:

```bash
sed -i '' "s/123456789012/$(aws sts get-caller-identity --query Account --output text)/" project/PvtAviation/agentcore/aws-targets.json
```

(On Linux, drop the empty `''` after `-i`.) `make deploy` rewrites the remaining files with real ARNs from your own account during the deploy chain (`config/state.json` from `deploy_lambdas.py`; `agentcore.json` and `harness.json` from the AgentCore CLI commands the Makefile invokes).

## Regenerate local manifests

The committed project scaffold under `project/PvtAviation/` is a redacted snapshot. To regenerate it from scratch against your own account, delete `project/` and run `make project` (or the full `make deploy`). The Makefile calls `agentcore create`, `agentcore add gateway`, `agentcore add gateway-target`, and `agentcore add tool` in sequence, which writes a fresh `agentcore.json`, `harness.json`, and CDK stack populated with your account's ARNs.

After regeneration, confirm `agentcore.json` has `"memories": []` and that `harness.json` has no `"memory"` field before running `make deploy-gateway`. The sample does not use memory.

## Deploy

```bash
make deploy
```

The chain runs in two phases:

**Phase A — Lambdas and project scaffold**
1. **IAM** — boto3 creates the Lambda execution role.
2. **Lambdas** — boto3 zips and deploys the three functions.
3. **Project** — `agentcore create --model-provider bedrock --model-id ...`
   scaffolds the harness project into `project/PvtAviation/`, and
   `agentcore add gateway` declares the MCP gateway in the project config.
4. **aws-targets.json** — populated with the current account and region;
   `npm install` runs inside the generated CDK project.
5. **First `agentcore deploy`** — runs the CDK stack (IAM execution role,
   Gateway) and then calls `CreateGateway` / `CreateHarness` in the
   post-CDK phase.

**Phase B — Wire Lambdas to the Gateway**
6. **Gateway targets** — `agentcore add gateway-target` registers each
   Lambda (three calls, one per tool) with the tool schema.
7. **Harness tool** — `agentcore add tool --type agentcore_gateway`
   attaches the gateway to the harness.
8. **Second `agentcore deploy`** — syncs the new targets and tool wiring
   to the live Gateway and Harness.

Individual stages are addressable: `make lambda-role`, `make lambdas`,
`make project`, `make aws-targets`, `make deploy-gateway`,
`make gateway-targets`, `make tool`, `make deploy-final`.

## Invoke

```bash
make invoke
```

Sends the demo booking prompt. The managed Harness loop dispatches through
the Gateway to the three Lambdas and streams the response, including
`🔧 Tool: <name>` markers for each call.

Per-call overrides are one flag change. For example, to switch the model for a single invocation:

```bash
cd project/PvtAviation && \
  agentcore invoke --harness PvtAviation \
    --model-id global.anthropic.claude-haiku-4-5 \
    "Same prompt, different model"
```

## Clean up

AgentCore CLI v1.0.0-preview does not ship an `agentcore destroy`
subcommand yet, so teardown is a three-step manual process. Run the steps
in order:

### 1. Delete the Harness (control-plane API)

```bash
HARNESS_ID=$(aws bedrock-agentcore-control list-harnesses \
  --region us-east-1 \
  --query "items[?contains(harnessName,'PvtAviation')].harnessId" \
  --output text)

aws bedrock-agentcore-control delete-harness \
  --harness-id "$HARNESS_ID" \
  --region us-east-1
```

### 2. Delete the Gateway (and its targets)

```bash
GATEWAY_ID=$(aws bedrock-agentcore-control list-gateways \
  --region us-east-1 \
  --query "items[?contains(name,'aviation-tools')].gatewayId" \
  --output text)

# Delete targets first — the gateway refuses to drop while targets exist
for TARGET_ID in $(aws bedrock-agentcore-control list-gateway-targets \
    --gateway-identifier "$GATEWAY_ID" --region us-east-1 \
    --query "items[].targetId" --output text); do
  aws bedrock-agentcore-control delete-gateway-target \
    --gateway-identifier "$GATEWAY_ID" \
    --target-id "$TARGET_ID" --region us-east-1
done

aws bedrock-agentcore-control delete-gateway \
  --gateway-identifier "$GATEWAY_ID" \
  --region us-east-1
```

### 3. Remove the CDK stack and the Lambdas

```bash
# IAM execution role and container build pipeline
( cd project/PvtAviation/agentcore/cdk && npx cdk destroy --force )

# Lambda tool backends and the Lambda execution role owned by this repo
make destroy
```

Run `make clean` at the end to remove `config/state.json`, or
`rm -rf project/` to remove the generated project entirely.

## Differences vs. `1-bedrock-agents/`

| Concern | `1-bedrock-agents/` | `3-agentcore-harness/` |
|---|---|---|
| Provisioning surface | AWS CDK (stable constructs) | AgentCore CLI (scaffold + CDK + post-CDK Harness API) |
| Agent resource | `CfnAgent` (action group → Lambda) | Harness resource (`harness.json` + `CreateHarness`) |
| Tool schema format | `functionSchema` on the action group | MCP tool schema array per target |
| Lambda event envelope | Bedrock action-group payload | Flat arguments on `event` (Gateway MCP shape) |
| Invocation | `bedrock-agent-runtime.invoke_agent` | `agentcore invoke --harness` (or `bedrock-agentcore.invoke_harness`) |
| Session parameter | `sessionId` (any string) | `runtimeSessionId` (>= 33 chars, UUID-like) |
| Model binding | Fixed at agent creation | Default in `harness.json`, override per call |
| Tool binding | Fixed at action-group creation | Default in `harness.json`, override per call |
